# terraform-provider-bridgeport

The official Terraform/OpenTofu provider for [BridgePort](https://github.com/bridgeinpt/bridgeport). It manages BridgePort **configuration** declaratively via the platform's HTTP API.

This is a **separate repo** from the platform (`bridgeinpt/bridgeport`) because the Terraform and OpenTofu registries hard-require the `terraform-provider-*` repo naming and consume that repo's GitHub releases, and because the provider needs its own semver line (a provider release supports a *range* of platform versions). The contract that keeps the two in lockstep â€” the typed OpenAPI spec, the API stability policy, the Go SDK, and a provider-compat CI job â€” lives in the platform repo. See the epic: `bridgeinpt/bridgeport#197`.

## Architecture

- **Plugin framework**: [`terraform-plugin-framework`](https://developer.hashicorp.com/terraform/plugin/framework) (protocol v6), not the legacy SDKv2.
- **`main.go`** serves the provider; `version` is injected at build time via `-ldflags` by GoReleaser.
- **`internal/provider/`** holds everything:
  - `provider.go` â€” provider schema (`endpoint`, `token`), `Configure()` builds the SDK client and validates the token, and registers data sources / resources.
  - `*_data_source.go` / `*_resource.go` â€” one file per data source / resource.
- **The SDK does the HTTP.** All API calls go through `github.com/bridgeinpt/bridgeport/client`; the provider never hand-rolls requests. The SDK is **read-only today**, which is why the initial release ships data sources only.

## Design tenets (do not violate)

These mirror the platform epic and are the reason the provider is safe to adopt:

1. **Configuration only â€” runtime stays imperative.** Never model deploys, restarts, rollbacks, or live status as managed resources. Runtime fields (`status`, `exposedPorts`, health, discovery) are `Computed`, never required/optional inputs. The platform's `READONLY_FIELD` registry maps 1:1 to this split.
2. **Secrets never enter state.** Secret *values* are write-only arguments paired with a `version`/rotation attribute. Never read a secret value back into state.
3. **`plan` works offline.** No live API calls during plan/validate. Diff against submitted configuration, not against live runtime state.
4. **Natural keys for import.** `terraform import` uses `environment` + `name`/`key`, matching the API.

## Conventions

- **Data source / resource naming**: `bridgeport_<noun>` (singular = lookup by key, plural = list). `Metadata()` sets `resp.TypeName = req.ProviderTypeName + "_<noun>"`.
- **Models**: a `tfsdk:"..."` struct per data source/resource. Map SDK structs to models in a small helper (see `environmentToModel`). Snake_case attribute names.
- **Configure type assertion**: every data source/resource asserts `req.ProviderData.(*client.Client)` and emits a clear diagnostic on mismatch (guards against a nil/early Configure).
- **Diagnostics over panics**: surface API errors via `resp.Diagnostics.AddError` with the endpoint and underlying error; never `log.Fatal` outside `main`.
- **Docs are generated, not hand-edited after the fact.** Edit schema `MarkdownDescription` and `examples/`, then run `make generate` (tfplugindocs) to refresh `docs/`. The PR template checks this.

## Adding a data source or resource

1. Add `internal/provider/<noun>_data_source.go` (or `_resource.go`). Implement the framework interfaces; assert the client in `Configure`.
2. Register the factory in `provider.go` (`DataSources()` / `Resources()`).
3. Add an example under `examples/{data-sources,resources}/bridgeport_<noun>/`.
4. Add an acceptance test (`<noun>_data_source_test.go`) gated by `TF_ACC`, using `testAccProtoV6ProviderFactories` + `testAccPreCheck`.
5. `make generate` to refresh `docs/`. `make testacc` to verify.
6. If the data source/resource needs API surface the SDK doesn't expose yet, **add it to the SDK first** (`client/` in the platform repo) and release it â€” see below.

## SDK / API coupling

- The provider depends on a **tagged or pseudo-versioned** SDK module. Any provider feature that needs new request/response shapes requires a corresponding SDK change in the platform repo, released as a `client/vX.Y.Z` tag, then bumped here in the same PR.
- API changes upstream are governed by the [API Stability Policy](https://github.com/bridgeinpt/bridgeport/blob/master/docs/api-stability.md). Breaking wire changes ship only in a platform major; the provider must keep working across the supported range or document a minimum BridgePort version.

## Testing

- **Unit** (`make test`): fast, no live instance.
- **Acceptance** (`make testacc`, `TF_ACC=1`): runs against a **disposable** BridgePort spun up by `scripts/acc-harness.sh` (docker run â†’ mint admin token â†’ run suite â†’ tear down). Override the image with `BRIDGEPORT_IMAGE` (default `ghcr.io/bridgeinpt/bridgeport:edge`).
- CI: `.github/workflows/test.yml` runs build/vet/lint/unit + acceptance on every PR using **GitHub-owned actions only**, so it works under a restrictive org "allowed actions" policy. Don't add third-party actions to `test.yml` without confirming they're allowlisted.

## Releasing

- Tag `vX.Y.Z` â†’ `.github/workflows/release.yml` runs GoReleaser (cross-compile, zips, `SHA256SUMS`, **GPG-signed** checksums) and publishes a GitHub Release in the layout the registries ingest.
- Prerequisites (one-time): repo secrets `GPG_PRIVATE_KEY` + `PASSPHRASE`, and the `goreleaser/goreleaser-action` + `crazy-max/ghaction-import-gpg` actions allowlisted. See the repo's `MANUAL_SETUP.md` (or the bootstrap handoff).
